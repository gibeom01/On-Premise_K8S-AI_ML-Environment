# 테스트 로드맵

1. 100% 테스트 가능한 영역 (논리적 인프라 및 자동화)
물리적인 연산 장치(GPU) 없이도, 인프라를 코드로 구성하고 플랫폼을 올리는 핵심 아키텍처링은 모두 구현할 수 있습니다.

IaC 기반 인프라 프로비저닝 (Terraform & Ansible)

Terraform: vsphere 프로바이더를 사용하여 ESXi 위에 마스터 노드용 VM과 워커 노드용 VM, 스토리지용 VM을 자동으로 생성하고 네트워크를 구성하는 코드를 작성하고 테스트할 수 있습니다.

Ansible: 생성된 VM들에 SSH로 접근하여 OS 기초 설정, Docker/containerd 설치, 쿠버네티스 사전 요구사항(Swap off 등)을 구성하는 플레이북을 완벽하게 검증할 수 있습니다.

Kubernetes 클러스터 구축 및 관리

kubeadm이나 kubespray를 이용해 실제 운영 환경과 동일한 형태의 멀티 노드 쿠버네티스 클러스터를 VM 기반으로 구축할 수 있습니다.

Calico나 Cilium 같은 CNI(Container Network Interface) 플러그인 설정도 모두 동일하게 진행됩니다.

데이터 파이프라인 및 MLOps 플랫폼 배포

클러스터 내부에 MinIO(오브젝트 스토리지), Kafka(메시지 큐)를 컨테이너로 배포하여 데이터 레이크의 형태를 구성할 수 있습니다.

Kubeflow 플랫폼 전체를 배포하고, ArgoCD를 연동하여 CI/CD 파이프라인을 구축하는 것 역시 아무런 제약 없이 테스트 가능합니다.

관측성(Observability) 체계 구축

Prometheus와 Grafana를 설치하여 노드(VM)의 CPU, 메모리, 네트워크 트래픽을 모니터링하는 체계를 구성할 수 있습니다.

2. 제한적으로 테스트 가능한 영역 (Workaround 필요)
실제 대규모 데이터 연산이 들어가는 부분은 기능의 ‘흐름’만 검증하는 방식으로 우회해야 합니다.

GPU 기반 모델 학습 및 스케줄링

ESXi 호스트에 실제 물리 GPU가 꽂혀 있다면, PCIe Passthrough 기능을 이용해 특정 VM에 통째로 할당하여 nvidia-device-plugin과 드라이버 설치를 테스트해 볼 수 있습니다. (단, vGPU로 쪼개는 것은 엔터프라이즈 라이선스가 필요합니다.)

GPU가 전혀 없다면, Kubeflow Pipelines(KFP)에서 모델을 학습시킬 때 CPU 기반 학습으로 설정하거나 아주 가벼운 샘플 데이터만 태워, "데이터 전처리 → 파드 생성(TFJob/PyTorchJob) → 모델 저장"이라는 파이프라인의 파드 오케스트레이션 흐름 자체를 검증하는 데 집중할 수 있습니다.

3. ESXi VM 환경에서 테스트 불가능한 영역
물리 서버(Baremetal) OOB 관리 및 펌웨어 제어

예를 들어, 실제 물리 환경에서는 자동화 스크립트를 통해 Dell R620은 iDRAC7, R630은 iDRAC8, R640은 iDRAC9 버전에 맞춰 펌웨어를 제어하고 RAID를 묶는 작업이 필요하지만, 가상화된 ESXi 환경 내부에서는 이러한 OOB(Out-of-Band) 관리 인터페이스 통신을 테스트할 수 없습니다.

고성능 스토리지 및 네트워크 I/O 성능 테스트

NVMe-oF 브릿지 구성이나 InfiniBand 기반의 RDMA 네트워크 테스트는 물리적인 하드웨어 스위치와 NIC가 필요하므로 가상 환경에서는 구성이 어렵습니다.
